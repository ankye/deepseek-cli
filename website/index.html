🔎 Evidence-First Product Guarantee

DeepSeek CLI's evidence-first guarantee means that for fact-sensitive tasks — repository facts, product copy, commands, architecture, roadmap, release state, or evaluation — the workflow must gather local project evidence before producing factual output.

1

Evidence Plan

Why evidence is required, candidate sources, required fact classes, minimum source coverage, stop conditions.

2

Evidence Discovery

Inspect local project sources (README, package.json, source code, docs, OpenSpecs, tests) with bounded previews and fingerprints.

3

Claim Classification

Every factual claim is classified as verified, inferred, assumption, or unsupported.

4

Grounded Output

Generated artifacts, product pages, release notes, and evaluation reports link claims to evidence. Unsupported claims are removed or marked.

This very webpage was built using the evidence-first workflow. See website/evidence.json for the full evidence manifest, claim groundings, and assumptions.

🗺️ Product Roadmap / 产品路线图

DeepSeek follows a CLI-first product route: make the terminal CLI the first polished daily-use surface, then promote proven workflows to other hosts through shared protocol events.

🥇
CLI is the default near-term product surface

Interaction, approvals, diagnostics, extension management, and release readiness.

🔌
Non-CLI hosts stay as adapter skeletons

Until the corresponding CLI workflow passes acceptance gates.

📋
Promotion requires evidence

CLI smoke evidence, protocol fixtures, policy/audit traces, deterministic tests, and documented CLI behavior.

🏛️
CLI UX consumes shared runtime

CLI polish must not bypass runtime ownership — it consumes shared runtime events, command results, readiness results, and policy decisions.

🏛️ Platform Principles / 平台原则

🏛️

One Kernel, Many Hosts

🧪

One Quality System
