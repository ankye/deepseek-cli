## Context

The acceptance index is generated by `scripts/write-acceptance-index.mjs` and is used as a lightweight map from release gates to verification commands or test suites. Recent work added activation evidence gating and text-mode explanations, but the index has only one broad "Index provider CLI safety" row.

Acceptance index 由 `scripts/write-acceptance-index.mjs` 生成，用于把 release gates 映射到 verification commands 或 test suites。最近已经加入 activation evidence gating 与 text-mode explanations，但 index 目前只有一个宽泛的 "Index provider CLI safety" 行。

## Goals / Non-Goals

**Goals:**
- Make provider activation evidence and text evidence rendering visible in acceptance evidence.
- Point both gates to existing e2e/contract suites rather than duplicating checks.
- Keep the generator as the source of truth for `acceptance-index.md`.

**Non-Goals:**
- Run or refresh every `tests/acceptance/latest/*` artifact in this slice.
- Add new CLI commands or release-readiness DTOs.
- Change npm publishing behavior.

## Decisions

- Add separate index rows instead of expanding one generic row.
  - Rationale: provider intent safety, activation evidence gating, and text evidence rendering are related but distinct release risks.
  - Alternative considered: keep one row with a longer description. That is less searchable and easier to miss during release review.

- Reuse existing e2e and contract tests as evidence.
  - Rationale: `tests/e2e/local-readiness-cli.test.ts`, `src/apps/cli/test/cli.test.ts`, and `tests/contracts/index-provider-contracts.test.ts` already cover the behavior.

## Risks / Trade-offs

- [Risk] The acceptance index can drift if manually edited. -> Mitigation: update the generator and regenerate the file.
- [Risk] Evidence paths point to broad latest artifacts. -> Mitigation: include concrete test file names in the command/evidence column.
